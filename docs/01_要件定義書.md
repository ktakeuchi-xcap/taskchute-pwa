## 1. プロジェクト概要

タスクシュート方式によるタスク管理ツール「Taskchute（GAS 版）」を、モダンな技術スタックの PWA として全面リプレースする。Google Workspace の各種サービス（Sheets / Calendar / Tasks）との連携を踏襲しつつ、操作性・保守性・拡張性を大幅に向上させる。

---

## 2. 背景

### 2.1 現行システムの構成

- Google Apps Script（GAS）で実装
- Google Sheets を DB として利用
- サイドバー UI（PC 利用）＋ WebApp UI（モバイル利用）
- Google Calendar / Google Tasks と双方向同期

### 2.2 現行の課題

| ID | 課題 | 影響 |
|----|------|------|
| P-01 | GAS のコールドスタートが遅い（サイドバー起動に5〜10秒） | 起動のたびに集中が途切れる |
| P-02 | WebApp.html のスクリプトが未実装で機能しない | モバイル操作が事実上できない |
| P-03 | 型安全性がなく、`headers.indexOf()` の戻り値が `-1` の場合の保護がない | サイレントバグの温床 |
| P-04 | UTC↔JST 変換のワークアラウンドがコードに散在（`setDate(+1)` 等） | 日付ロジックが壊れやすい |
| P-05 | `syncWaitingList()` でループ中に行削除しているためインデックスがズレうる | データ破損リスク |
| P-06 | テストコードが書けず、リファクタが困難 | 機能追加のたびに不安が伴う |
| P-07 | スプレッドシート起動が前提のため、独立アプリとして使えない | デバイスを選ばない利用ができない |
| P-08 | リアルタイム性がない（タイマーや進捗表示なし） | タスクシュートの本質的価値が薄い |

## 3. 目的とゴール

### 3.1 目的

現行 GAS 版が抱える構造的問題を解消し、長期的に保守・拡張可能な個人タスク管理基盤を構築する。

### 3.2 ゴール（成功基準）

| ID | ゴール | 測定方法 |
|----|------|---------|
| G-01 | 起動時間を 2 秒以内に短縮 | Chrome DevTools の Lighthouse |
| G-02 | PC / モバイル双方で同一機能を提供 | 機能一覧チェックリスト |
| G-03 | TypeScript で型カバレッジ 100% | `tsc --noEmit` パス |
| G-04 | 主要機能の自動テストカバレッジ 80% 以上 | Vitest カバレッジレポート |
| G-05 | オフライン時もタスク閲覧と開始/終了が可能 | PWA オフラインテスト |
| G-06 | 金銭コスト ¥0 で運用 | 請求書ゼロ |

---

## 4. スコープ

### 4.1 In Scope

- React + TypeScript ベースの PWA 開発
- Google Sheets / Calendar / Tasks API との直接連携
- OAuth 2.0 認証
- 既存スプレッドシート構造の継承（後方互換）
- 単体・結合・E2E テスト
- GitHub Pages または Cloudflare Pages へのデプロイ
- GAS 版からの段階的移行

### 4.2 Out of Scope

- マルチユーザー対応（個人利用のみ）
- バックエンドサーバーの構築
- ネイティブアプリ（iOS/Android）のストア配布
- ガントチャート機能（別プロジェクトの `gantt_app_template` で対応）
- Slack / Outlook 等他サービスとの連携（将来検討）
- レポート・分析機能（将来検討）

## 7. 非機能要件

### 7.1 性能

| ID | 要件 | 目標値 |
|----|------|--------|
| NF-P01 | 初回起動時間（キャッシュなし） | 3 秒以内 |
| NF-P02 | 2 回目以降の起動時間 | 1 秒以内 |
| NF-P03 | タスク追加〜画面反映 | 200ms 以内（楽観的更新） |
| NF-P04 | タスク一覧の表示 | 500ms 以内 |
| NF-P05 | バックグラウンド同期 | 30 秒以内に完了 |

### 7.2 可用性

| ID | 要件 |
|----|------|
| NF-A01 | Google サービスがダウン中もタスク閲覧と開始/終了は可能 |
| NF-A02 | オフライン時の操作は再接続時に自動同期 |

### 7.3 セキュリティ

| ID | 要件 |
|----|------|
| NF-S01 | OAuth 2.0 (Google Identity Services) で認証 |
| NF-S02 | アクセストークンは sessionStorage に保管（永続化しない） |
| NF-S03 | スコープは最小権限の原則（必要なシート・カレンダーのみ） |
| NF-S04 | OAuth クライアント ID 以外の機密情報をリポジトリに含めない |
| NF-S05 | HTTPS でのみ動作（GitHub Pages / Cloudflare Pages は標準対応） |

### 7.4 ユーザビリティ

| ID | 要件 |
|----|------|
| NF-U01 | スマートフォン（iPhone / Android）で快適に操作可能 |
| NF-U02 | キーボード操作（タブ移動・Enter送信）に対応 |
| NF-U03 | 主要操作は 1〜2 タップ以内で完了 |
| NF-U04 | エラーメッセージは具体的な行動を促す表現 |

### 7.5 保守性

| ID | 要件 |
|----|------|
| NF-M01 | TypeScript で全モジュールを記述 |
| NF-M02 | 機能単位でディレクトリを分割（feature-based） |
| NF-M03 | Lint / Format / 型チェックを CI で自動実行 |
| NF-M04 | 主要関数に単体テスト |

### 7.6 移植性・拡張性

| ID | 要件 |
|----|------|
| NF-X01 | データ層を抽象化し、将来 IndexedDB のみ運用へ移行可能 |
| NF-X02 | Google サービス以外（例：Notion）への切替が現実的なコストで可能 |

## 8. 制約条件

| ID | 制約 | 理由 |
|----|------|------|
| C-01 | サーバー・有料 SaaS を一切使わない | ゼロコスト要件 |
| C-02 | データの正本は既存 Google Sheets を継続利用 | 移行リスク最小化、GAS 版との並行稼働 |
| C-03 | 既存シートのスキーマを破壊的に変更しない | 同上 |
| C-04 | 公開する OAuth クライアント ID は機密扱いしない（PKCE 前提） | SPA の構造上必然 |
| C-05 | デプロイ先は無料枠ホスティングのみ（GitHub Pages / Cloudflare Pages 等） | コスト要件 |

## 10. 用語集

| 用語 | 定義 |
|------|------|
| タスクシュート | タスクを時系列に並べ、開始・終了時刻を実績記録する時間管理手法 |
| TaskDB | 通常タスクを記録するスプレッドシートシート |
| WaitingList | 確認待ちタスクを記録するスプレッドシートシート |
| RoutineTasks | ルーチンタスクの定義シート |
| Settings | カテゴリマスタを保持するシート |
| 確認待ちタスク | 他者の応答を待っているタスク（Google ToDo と連携） |
| ルーチンタスク | 毎日・毎週など定期的に発生するタスク |
| PWA | Progressive Web App。ブラウザでアプリ的に動作するWeb技術群 |
| PKCE | Proof Key for Code Exchange。バックエンドなしのOAuth認証手法 |
| 楽観的更新 | サーバー応答を待たずにUIを先に更新するパターン |
