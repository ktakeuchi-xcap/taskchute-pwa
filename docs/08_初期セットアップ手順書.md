# 初期セットアップ手順書（新しく使い始める方向け）

| 項目 | 内容 |
|------|------|
| 対象 | 竹内以外の人が、自分専用のデータでTaskchute PWAを新しく使い始める場合 |
| 所要時間 | 30〜60分程度 |
| 前提 | Googleアカウント、GitHubアカウント |

---

## まず知っておくこと

このアプリは、**1回のビルド（デプロイ）ごとに、参照するスプレッドシートとカレンダーが1組に固定される**仕組みになっている。ログイン画面でGoogleアカウントを切り替えても、参照先のスプレッドシート・カレンダーは変わらない。

そのため、**自分専用のデータで使いたい人は、それぞれ自分専用のデプロイ（＝自分のGitHubリポジトリとGitHub Pages）を持つ必要がある**。既存のアプリのURLにアクセスして自分のGoogleアカウントでログインするだけでは、竹内のスプレッドシートを見にいってしまい、権限がなければエラーになる。

準備するものは大きく4つ。

| # | 準備するもの | 内容 |
|---|--------------|------|
| 1 | スプレッドシート | 自分専用の新規スプレッドシート。決まった構成のシート（タブ）を用意する |
| 2 | カレンダー | タスクを予定として登録するGoogleカレンダー |
| 3 | OAuthクライアントID | Googleへのログインを許可するための認証情報 |
| 4 | デプロイ環境 | 上記3つを紐付けた、自分専用のアプリ本体（GitHub Pages） |

---

## 手順1：スプレッドシートを用意する

Google Sheetsで新規スプレッドシートを1つ作成する。名前は任意（例：「Taskchute」）。

作成したら、以下のシート（タブ）を用意する。**シート名・列見出しは記載の通り正確に入力する**（大文字小文字も区別される）。列の順番は自由（見出し名で参照するため）。

### 1-1. `TaskDB`（必須）

タスクと会議の本体データを保存するシート。1行目に見出し行を入れる。

| 列見出し | 必須 | 内容 |
|----------|------|------|
| `TaskID` | ○ | タスクの一意ID（アプリが自動採番） |
| `TaskName` | ○ | タスク名 |
| `Category` | | 案件名 |
| `EstimateMinutes` | ○ | 見積時間（分） |
| `ScheduledStartTime` | ○ | 予定開始時刻 |
| `ScheduledEndTime` | ○ | 予定終了時刻 |
| `ActualStartTime` | | 実績開始時刻 |
| `ActualEndTime` | | 実績終了時刻 |
| `Status` | ○ | `Not Started` / `In Progress` / `Done` |
| `CalendarEventID` | ○ | 紐付くカレンダーイベントのID |

以下は後から機能を使う場合に追加する任意の列（無くてもアプリは動く。その機能だけ使えない）。

| 列見出し | 必須 | 内容・追加すると使えるようになる機能 |
|----------|------|--------------------------------------|
| `Source` | 任意 | 会議の自動取り込み機能。無いと会議は同期されない |
| `RecurringEventID` | 任意 | 定例会議への案件タグ付け「これ以降」「すべて」機能 |
| `CountsTowardWorkload` | 任意 | 「工数に計上する/しない」設定機能。無いと常に計上する扱い |

> `CountsTowardWorkload`列にGoogleスプレッドシートの「チェックボックス」形式を設定して運用しても問題ない（チェック＝計上する、未チェック＝計上しない、として正しく認識される）。

### 1-2. `Settings`（必須）

案件（カテゴリ）マスタを管理するシート。1行目は見出し行（内容は自由、例：「案件名」「色」）、2行目以降にデータを入れる。

| 列 | 内容 |
|----|------|
| A列 | 案件名 |
| B列 | 表示色（16進カラーコード、例：`#4F46E5`）。空欄でも可 |

案件はアプリの設定画面からも追加・編集できるため、最初は空（見出し行のみ）で始めてよい。

> このシートの`Z1`セルはアプリが同期の排他制御に内部利用する。手を加えず空のままにしておく。

### 1-3. `RoutineTasks`（必須）

毎日・毎週などのルーチンタスクを自動生成する機能で使う。1行目に見出し行を入れる（ルーチンを使わない場合も、見出し行だけは作っておく）。

| 列見出し | 内容 |
|----------|------|
| `Schedule` | `毎日` / `月`〜`日`（曜日） / `初日` / `末日` / `15日`のような日付指定 |
| `TaskName` | タスク名 |
| `StartTime` | 開始時刻（`HH:mm`） |
| `Category` | 案件名 |
| `EstimateMinutes` | 見積時間（分） |

### 1-4. `WaitingList`（必須）

「確認待ち」タスク（Google ToDoと連携）で使う。1行目に見出し行を入れる（使わない場合も見出し行だけは作っておく）。

| 列見出し | 内容 |
|----------|------|
| `SystemTaskID` | 内部ID（アプリが自動採番） |
| `TaskName` | 依頼内容 |
| `WaitingFor` | 依頼先 |
| `DelegatedDate` | 依頼日 |
| `FollowUpDate` | フォローアップ予定日 |
| `GoogleTaskID` | 紐付くGoogle ToDoのID |

### 1-5. `MeetingCategoryRules`（任意）

会議を「これ以降のすべての予定」「すべての予定」という範囲で案件タグ付け・工数対象外設定をしたい場合にのみ必要。使わないなら作らなくてよい（「この予定のみ」の設定は無くても動く）。

| 列見出し | 内容 |
|----------|------|
| `RecurringEventID` | 定例会議のシリーズID |
| `Category` | 案件名 |
| `EffectiveFromDate` | この設定を適用し始める日付（空欄なら全期間） |
| `CountsTowardWorkload` | 工数に計上するか（チェックボックス可） |
| `WorkloadEffectiveFromDate` | 工数設定を適用し始める日付（空欄なら全期間） |

作成したら、スプレッドシートのURLから **スプレッドシートID** を控えておく。

```
https://docs.google.com/spreadsheets/d/【ここがID】/edit
```

## 手順2：カレンダーを用意する

### 2-1. タスク用カレンダー（必須）

タスクを「開始〜終了」の予定としてイベント登録する先のカレンダー。既存のGoogleカレンダー（メインカレンダーなど）をそのまま使ってもよいし、専用の新規カレンダーを作ってもよい。

カレンダーIDの調べ方：Googleカレンダーの対象カレンダーの設定 →「カレンダーの統合」→「カレンダーID」。メインカレンダーの場合はGoogleアカウントのメールアドレスがそのままIDになる。

### 2-2. 会議取り込み用カレンダー（任意）

既存の会議予定を「予定」「実績」画面に自動で取り込みたい場合、そのカレンダーのIDを控えておく。多くの場合、タスク用カレンダーと同じ（自分のメインカレンダー）でよい。使わない場合はこの手順は不要。

---

## 手順3：OAuthクライアントIDを用意する

Googleアカウントでのログインを許可するための認証情報。詳しい手順は [docs/07_認証セットアップガイド.md](./07_認証セットアップガイド.md) を参照（Google Cloud Consoleでの操作を10分程度で解説）。

判断の目安：

| 状況 | 対応 |
|------|------|
| 竹内と同じGoogle Workspace組織（xcap.co.jp）に所属している | 既存のOAuthクライアントに、自分のデプロイ先URLを「承認済みのJavaScript生成元」として追加してもらえば流用できる可能性がある（竹内に相談） |
| 組織外、または独立して管理したい | docs/07の手順に沿って自分専用のOAuthクライアントIDを新規発行する |

いずれの場合も、発行・追加された **クライアントID** を控えておく。

## 手順4：アプリをデプロイする

1. このリポジトリを自分のGitHubアカウントにフォークする
2. フォークしたリポジトリの **Settings → Secrets and variables → Actions → Variables** に、以下の4つを登録する（手順1・2・3で控えた値）

   | 変数名 | 値 |
   |--------|-----|
   | `GOOGLE_OAUTH_CLIENT_ID` | 手順3のクライアントID |
   | `TASKCHUTE_SPREADSHEET_ID` | 手順1のスプレッドシートID |
   | `TASKCHUTE_CALENDAR_ID` | 手順2-1のカレンダーID |
   | `MEETING_CALENDAR_ID` | 手順2-2のカレンダーID（使わない場合は空欄のままでよい） |

3. **Settings → Pages** で、Source を「GitHub Actions」に設定する
4. `main`ブランチに何かpush（またはActionsタブから`Deploy to GitHub Pages`ワークフローを手動実行）すると、自動的にビルド・デプロイされる
5. デプロイ完了後、`https://【GitHubユーザー名】.github.io/【リポジトリ名】/` でアプリにアクセスできる

> このURLを、手順3で発行したOAuthクライアントの「承認済みのJavaScript生成元」に追加するのを忘れないこと（追加していないとログイン時にエラーになる）。

---

## 手順5：初回ログイン・動作確認

1. デプロイされたURLをブラウザで開く
2. 「Googleでサインイン」を押し、手順1で紐付けたGoogleアカウントでログインする
3. 案件・タスクが正しく表示される（最初は空でよい）ことを確認する
4. 試しにタスクを1件追加し、スプレッドシートの`TaskDB`シートとGoogleカレンダーの両方に反映されることを確認する

うまくいかない場合は [docs/07_認証セットアップガイド.md](./07_認証セットアップガイド.md) の「トラブルシューティング」の章を参照。
