## 1. システム構成

### 1.1 全体図

```mermaid
flowchart LR
    subgraph Client["クライアント（ブラウザ / PWA）"]
        UI["React UI Layer"]
        Hooks["TanStack Query / Zustand"]
        Cache[("IndexedDB<br/>キャッシュ・<br/>書き込みキュー")]
        SW["Service Worker"]
        UI --> Hooks
        Hooks --> Cache
        SW --> Cache
    end

    subgraph Google["Google Services"]
        Sheets["Sheets API v4<br/>(データベース)"]
        Cal["Calendar API v3<br/>(タスクイベント)"]
        Tasks["Tasks API v1<br/>(確認待ち)"]
        GIS["Identity Services<br/>(認証)"]
    end

    subgraph Hosting["静的ホスティング (無料)"]
        Pages["GitHub Pages / Cloudflare Pages"]
    end

    Hooks -.OAuth.-> GIS
    Hooks -.HTTPS.-> Sheets
    Hooks -.HTTPS.-> Cal
    Hooks -.HTTPS.-> Tasks
    Browser["ブラウザ"] -.初回ロード.-> Pages
    Pages -.静的アセット.-> Browser
```

### 1.2 サーバー構成

**バックエンドサーバーは存在しない。** 静的ファイルホスティングのみ。すべてのビジネスロジックはブラウザ上で実行され、データの永続化は Google サービス（Sheets / Calendar / Tasks）に委譲する。

## 4. 認証アーキテクチャ

### 4.1 採用方式

**Google Identity Services（GIS）のトークンモデル** を採用。Implicit Grant 相当だが、ポップアップ／iframe 経由の silent renewal をサポートする現代的な方式。

### 4.2 認証フロー（初回）

```mermaid
sequenceDiagram
    actor User
    participant App as Taskchute PWA
    participant GIS as Google Identity Services
    participant API as Google APIs

    User->>App: アプリ起動
    App->>App: sessionStorage 確認
    App->>GIS: requestAccessToken({prompt:''})
    GIS-->>App: 認証済みなら token / 未認証なら失敗
    alt 未認証
        App->>User: 「ログイン」ボタン表示
        User->>App: ボタン押下
        App->>GIS: requestAccessToken({prompt:'consent'})
        GIS->>User: 同意画面（ポップアップ）
        User->>GIS: 同意
        GIS-->>App: access_token (1h)
    end
    App->>App: token を sessionStorage に保管
    App->>API: API 呼び出し（Bearer token）
    API-->>App: データ
```

### 4.3 トークン更新

- アクセストークン期限切れ（401）を検知したら自動的に `requestAccessToken({prompt:''})` で silent renewal を試行
- silent renewal に失敗した場合のみ明示的なログイン画面へ遷移

### 4.4 スコープ設計（最小権限）

| スコープ | 用途 |
|---------|------|
| `https://www.googleapis.com/auth/spreadsheets` | TaskDB / WaitingList / Settings / RoutineTasks の読み書き |
| `https://www.googleapis.com/auth/calendar.events` | Taskchute カレンダーのイベント操作（カレンダー一覧は不要） |
| `https://www.googleapis.com/auth/tasks` | 確認待ちタスクの読み書き |

`drive.metadata.readonly` 等の余剰スコープは要求しない。

## 5. データフロー

### 5.1 タスク追加フロー

```mermaid
sequenceDiagram
    participant U as User
    participant UI as React UI
    participant Q as TanStack Query
    participant IDB as IndexedDB
    participant GAPI as Google APIs

    U->>UI: 「タスク追加」ボタン押下
    UI->>Q: useAddTask.mutate(data)
    Q->>Q: 楽観的更新でキャッシュ即時反映
    UI-->>U: 画面更新（< 50ms）
    Q->>GAPI: Calendar.events.insert
    GAPI-->>Q: event.id
    Q->>GAPI: Sheets.append (TaskDB)
    GAPI-->>Q: ok
    Q->>IDB: ローカルキャッシュ更新
    alt 失敗
        Q->>Q: ロールバック
        UI-->>U: エラートースト
    end
```

### 5.2 カレンダー逆同期フロー（定期）

```mermaid
sequenceDiagram
    participant SW as Service Worker
    participant App as App
    participant GAPI as Google APIs
    participant IDB as IndexedDB

    Note over SW,App: 30秒ごとまたは起動時
    App->>GAPI: Calendar.events.list (±15日)
    GAPI-->>App: events[]
    App->>GAPI: Sheets.values.get (TaskDB)
    GAPI-->>App: rows[]
    App->>App: Diff (CalendarEventID で突合)
    App->>GAPI: Sheets.batchUpdate (変更分のみ)
    App->>IDB: ローカルキャッシュ更新
    App->>App: TanStack Query invalidate
```

## 6. オフライン戦略

### 6.1 読み取り

- IndexedDB に最新のタスク一覧をミラー保存
- オフライン時は IndexedDB から表示
- 接続復帰時に最新化

### 6.2 書き込み

- オフライン時の書き込み操作は IndexedDB の `mutation_queue` に積む
- Service Worker の Background Sync API で再接続時に自動フラッシュ
- フラッシュ中は順序保証（FIFO）

### 6.3 競合解決

| シナリオ | 解決方針 |
|---------|---------|
| 同一タスクの編集競合 | 最終更新時刻が新しい方を採用 |
| ローカルで開始 → クラウドで先に終了済み | クラウドを優先、ローカル変更を破棄しユーザーに通知 |
| ローカル追加 → 同期前に Calendar 側で同名イベント追加 | 別タスクとして扱う（CalendarEventID が異なるため） |

## 8. セキュリティ

| 項目 | 対策 |
|------|------|
| トークン保管 | sessionStorage（タブ閉じで破棄）。localStorage は使わない |
| XSS 対策 | React の自動エスケープに依存、`dangerouslySetInnerHTML` 全面禁止 |
| CSRF | SPA + Bearer Token のため対象外 |
| Content Security Policy | meta タグでスクリプトソースを Google ドメインに限定 |
| 依存パッケージ脆弱性 | `npm audit` を CI で実行、Dependabot 有効化 |
